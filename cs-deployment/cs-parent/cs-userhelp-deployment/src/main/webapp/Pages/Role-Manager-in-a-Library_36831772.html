Sirma Platform : Role Manager in a Library

Users with the role Manager in a library cannot be changed or removed on any of the objects part of the library. Their role of Manager should be changed on library level from the Administrator in order to be changed in the objects in the library.

  • Management Permissions section in the Manage Permissions screen
    • No Management Permissions section on library level
    • Management Permissions section on lower levels (objects in the library)
      • The permissions in the Management Permissions section could not be edited
  • The permissions in the Management Permissions section are with the highest priority
    • When the role Manager is changed for an user on library level, then his special and inherited permissions becomes valid.

Management Permissions section in the Manage Permissions screen

On the ontology class landing page (library details page) there is no section "Management Permissions".

The section "Management Permissions" is visible in the Manage Permissions screen for all children of a library. In this section are displayed all users with the role Manager set on the library level.

No Management Permissions section on library level

  • The user opens the permissions tab of a library details page (1).
    • If the user has edit permissions on the project then the action Edit Permissions is active.
  • There is no Management Permissions section on project level (2).

Management Permissions section on lower levels (objects in the library)

Any of the library objects have the following permission sections visible when tab Permissions is opened - Special Permissions, Inherited Permissions, Management Permissions.

  • The user opens the permissions tab of an object in the library (1).
    • If the user has edit permissions on the library then the action Edit Permissions is active.
    • There is Management Permissions section on object level (2). The user with the role Manager in the library is visible in the section (3).

The permissions in the Management Permissions section could not be edited
  • The user with edit permissions on the library clicks the button "Edit Permissions" and the permission sections are opened in edit mode (1).
  • The Management Permissions section on the object level (2) is not editable.

The permissions in the Management Permissions section are with the highest priority

Examples:

  1. If an user is with the role Manager on a library, and with Special permissions Consumer on an object from the library, then the special permissions on the object are not taken into account.
  2. If an user is with the role Manager on a library, and with Inherited permissions Consumer on a document part of on object in that library, then the inherited permissions on the document are not taken into account.
  • The user (1) opens the object details page (2) of an object part of the library where he/ she is with the role Manager (3) and with Special Permissions of Consumer on the object (4).
  • The user  has edit permissions on the object (5) (not for a consumer).
  • The user opens the list of available actions on the object (6-7). The list displays all actions for the role Manager (not the actions for a consumer).

When the role Manager is changed for an user on library level, then his special and inherited permissions becomes valid.

Example:

  • The administrator changes the Manager of the library where user (1) was with the role of Manager (see the screen above).
  • The user (1) who was Manager of the library and with Special permissions of a consumer on the object opens the object details page (2).
  • The Management Permissions for the object are given to the newly selected manager of the library (3). 
  • The Special permissions of a consumer (4), given to the user (1) (former manage of the library), are valid:
    • the user (1) has NO edit permissions on the object (5) because now he is with consumer permissions on the object
    • the list of available actions on the object are for a consumer on the object (6-7).
