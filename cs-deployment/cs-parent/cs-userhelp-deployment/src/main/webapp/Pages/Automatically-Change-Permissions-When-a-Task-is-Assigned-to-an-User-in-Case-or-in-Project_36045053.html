Sirma Platform : Automatically Change Permissions When a Task is Assigned to an User in Case or in Project

When a task is assigned in case/ project to an user who is not member of the project the system automatically gives the respective user special permissions (of Contributor/ Collaborator) on the case/ project and the permissions remain the same when the user completes his task.

  • Rules
  • Examples
    • A task is assigned in a case with inherited permissions to an user who is not member of the project.
    • A task is assigned in the project to an user who is not member of the project.
    • When an user with Special permissions completes his task in a case - his permissions are not removed
    • When the user is member of a project (has personal or group permissions on a project or case) then his role on the respective case or project remains the same, no matter if a task is assigned to him or not.

Rules

1. All other users in a project have "No permissions" or "Consumer" permissions.

  • If the user is not member of a project and a task is assigned to him on a case - he receives special permissions of Collaborator in the case, and Consumer in the project.
  • If the user is not member of a project and a task is assigned to him on the project - he receives special permissions of a Consumer on the project.
  • When the user completes his task in a case - his permissions on the case remain Collaborator, on the project remain Consumer.

2. All other users in a project have "Contributor" or "Collaborator" permissions.

  • If the user is not member of a project and a task is assigned to him on a case - he receives special permissions of Collaborator in the case, and Contributor or Collaborator permissions in the project, based on the permissions given to all other users.
  • If the user is not member of a project and a task is assigned to him on the project - he receives special permissions of Contributor or Collaborator permissions in the project, based on the permissions given to all other users
  • When the user completes his task in a case or project his role is not changed.

3. When the user is member of a project (has personal or group permissions on a project or case) then his role on the respective case or project remains the same, no matter if a task is assigned to him or not. For example:

  • If a user is with individual permissions on a project set to "No permissions" then, when a task is assigned to him, he will remain with no permissions both on the project and the case, but if the user follows the link in the notification email sent to him he will be able to preview his task and complete it.
  • If a user is with individual permissions on a project set to Consumer then he will remain with this permissions even when a task on the project or case is assigned to him and he competes it.

Note:  For POOLED TASK - the user who claims the task receives the corresponding permissions on the project/ case and his role on the project/ case remains when the task is completed.

Examples

A task is assigned in a case with inherited permissions to an user who is not member of the project.
  • The user creates a project (1-2) where all users not members of the project have No Permissions (3) on the project (do not see the project).

  • A case is created in the project. The permissions flag of the case is with value "Inherited" (2).
  • The manager/ collaborator creates a new task in the case (3-4).

  • The created task (1) is assigned to an user (2) who is not a project member (does not have permissions to work on the project).

  • The task assignee automatically receives permission of Consumer in the project (1-2).

  • When the case details page is opened (1)
    • The permissions flag of case is changed to "Inherited and Special" (2). 
    • The task assignee automatically receives in the Special Permissions section Collaborator permissions (3).
    • The same user (task assignee) in the Inherited permissions section is with Consumer (4) permissions (inherited from the project).

    • All objects part of the case (1), which are with permissions flag Inherited (2), inherit from the parent case the new permission given to the user (the task assignee) (3).
      • The permission flag of all objects with inherited permissions remains the same "Inherited" (despite the flag of the parent case has changes to "Special permissions")

A task is assigned in the project to an user who is not member of the project.
  • User relevant permissions creates a task on project level (1-2-3).

  • The task is assigned to an user who is not member of the project (with no permissions on the project) (1-2).

  • The task assignee automatically receives permission of Consumer in the project.

When an user with Special permissions completes his task in a case - his permissions are not removed
  • When the user completes his task in a case - his permissions on the case remain Collaborator, on the project remain Consumer.
When the user is member of a project (has personal or group permissions on a project or case) then his role on the respective case or project remains the same, no matter if a task is assigned to him or not.
  • In a case with Inherited permissions (1-2) the user Daniela Todorova  is with individual permissions permissions set to NO_PERMISSION (3). A task is created in the case and assigned to this user.

  • The user with No_Permissions on the case opens the task following the link in the notification email sent to her (1-2-3) and may complete it.

  • On the project/ case level the permissions of the user who is a member of the project (has individual permissions) is not changed (1-2-3-4) and will remain the same when the task is completed.
