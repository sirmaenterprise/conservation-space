Sirma Platform : Role Manager in Project and in Project Domain Objects

Users with the role Manager in a project can not be changed or removed on any of the objects part of the project.

The role of Manager should be changed on project level in order to be changed in the project children entities.

  • Management Permissions section in the Manage Permissions screen
    • No Management Permissions section on project level
    • Management Permissions section on lower levels
      • The permissions in the Management Permissions section could not be edited
  • The permissions in the Management Permissions section are with the highest priority
    • When the role Manager on project level is given to another user, then the special and inherited permissions (of the former manager) become valid

Management Permissions section in the Manage Permissions screen

In the section are displayed all users with the role Manager on project level (not on lower levels).

No Management Permissions section on project level

  • The user opens the permissions tab of a project (1).
    • If the user has edit permissions on the project then the action Edit Permissions is active.
  • There is no Management Permissions section on project level (2).

Management Permissions section on lower levels

Any of the objects part of the project have the following permission sections visible when tab Permissions is opened - Special Permissions, Inherited Permissions, Management Permissions.

  • The user opens the permissions tab of created case in the project (1).
    • If the user has edit permissions on the project then the action Edit Permissions is active.
    • There is Management Permissions section on case level (2). The user with the role Manager in the project is visible in the section (3).

The permissions in the Management Permissions section could not be edited
  • The user  with edit permissions on the case clicks the button "Edit Permissions" and the permission sections are opened in edit mode (1).
  • The Management Permissions section on the case level (2) is not editable.

The permissions in the Management Permissions section are with the highest priority

Example: If an user is with the role Manager on a project and with Special permissions Consumer on a case, part of the project, then the special permissions on the case are not taken into account.

  • The user (1) opens the case details page (2) of a case part of a project where he/ she is with the role Manager (3) on the project and with Special Permissions - Consumer in the case (4).
  • The user opens the list of available actions on the case (5-6). The list displays all actions for the role Manager (not the actions for a consumer).

When the role Manager on project level is given to another user, then the special and inherited permissions (of the former manager) become valid

  • The Manager of the project (1) opens the project details page (2) and edits the permissions. He selects another user (3) as Manager of the project and saves the changes (4).

  • The edited permissions are saved.

  • The user who was Manager of the project (1) opens the Case details page (2).
    • The Management Permissions for the case are given to the newly selected manager of the project (3). 
    • The Special permissions of a consumer, given to the user (4) (former manage of the project), are valid - the list of available actions is for a consumer on the case (5-6).
